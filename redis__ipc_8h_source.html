66 // Init should be called from each thread that will be doing redis IPC,
67 // with all threads getting same component name but unique thread name.
68 // Cleanup is provided to prevent a slow leak from turnover of short-lived
69 // threads that each call init to allocate a few per-thread variables.
70 //
71 // The thread name should indicate the purpose of the thread and be
72 // predictable, e.g. live-http-worker-3, rather than randomly generated or
73 // based on TID number, because it will be used to generate name of results
74 // queue -- and we want same results queues to be re-used if component gets
75 // restarted (avoid having to garbage-collect stale, abandoned redis queues)
76 //
77 // Cleanup is done based on thread ID so that main thread can clean up for
78 // terminated threads, as long as it is tracking their IDs. For a
79 // single-threaded process, tid == pid.
80

82 int redis_ipc_cleanup(pid_t tid);

85 // A component can send a command to any other component,
86 // but it can only receive commands from its own command queue(s).
87 //
88 // A component sending a command should supply NULL as subqueue unless
89 // it has multiple command queues defined in redis_ipc.conf,
90 // in which case it should supply one of those queues. The same goes
91 // for a component waiting for commands -- if it only has one command
92 // queue, subqueue should be NULL.
93 //
94 // The timeout field is in seconds, or use zero to indicate no timeout
95 // for blocking.
96 //
97 // When submitting a command, the name of submitter's results queue
98 // and a unique command ID will automatically be inserted into the command
99 // as fields "results_queue" and "command_id" respectively.
100 //
101 // The submitter will block on its results until result has been received
102 // or timeout has been exceeded. If a result with _different_ command ID
103 // is received, it will be logged as an error to submitter component's
104 // debug channel then freed, and the wait on results queue will restart
105 // (expiration time will be reset to original).
106

110 // After receiving and executing a command, the receiving component
111 // should submit a result. The original command is passed in as a parameter
112 // to provide the command ID and path to results queue. The command ID
113 // will automatically be added to the result object before pushing it to the
114 // result queue so that the command submitter will only get back the
115 // expected result (as opposed to a stale result, belonging to a command that
116 // took so long that submitter timed out before seeing the result).

127 // A component can only write a setting if it has been authorized by redis_ipc.conf,
128 // but it can read any setting.
129 //
130 // Each setting is a set of fields that are key-value pairs, where both field name and
131 // values are stored as strings (of course values could be JSON text if needed).
132 //
133 // A setting can be written or read in its entirety using the first pair of functions:
134 // when writing, the JSON object parameter can hold all the component's setting fields
135 // as key-value pairs (although it would also work with a smaller set of fields);
136 // when reading, the JSON object returned *will* hold all existing setting fields.
137 //
138 // A setting can also be written or read a single field at a time using the second
139 // pair of functions, which works on strings rather JSON hashes. If multiple setting
140 // fields are being accessed, consider accessing the full thing rather than multiple
141 // calls to individual fields -- it may be more efficient.

172 // Each component can only send event messages to its own event channel(s),
173 // but can subscribe to any (or all) event channels.
174 //
175 // A component sending an event should supply NULL as subchannel unless
176 // it has multiple subchannels defined in redis_ipc.conf,
177 // in which case it should supply one of those subchannels.
178 //
179 // A component subscribing to events can use NULL as component parameter
180 // to watch all event channels for all components. Or, to watch all events
181 // from a single component, use NULL for subchannel (as mentioned above,
182 // not all components even have subchannels).
183 //
184 // The send function will automatically append the following fields,
185 // plus the standard ones (timestamp, etc) to the event:
186 // channel (full channel name)
187 //
188 // The unsubscribe function stops watching all event channels.

195 // Each component will send debug messages to its own debug channel,
196 // but can subscribe to any (or all) debug channels. Use NULL as
197 // component to subscribe to all debug channels.
198 //
199 // Debug send can serve as drop-in replacement for printf-style logging,
200 // which is why it doesn't take a JSON object. The send function internally
201 // generates a json object containing following fields, plus the standard
202 // ones (timestamp, etc)
203 // message
204 // level
205 // channel (full channel name)
206 //
207 // When sending debug messages, low debug level indicates
208 // high message priority, since only messages of lower or equal level
209 // to configured component debug verbosity will actually get sent.
210 //
211 // The unsubscribe function stops watching all debug channels.
